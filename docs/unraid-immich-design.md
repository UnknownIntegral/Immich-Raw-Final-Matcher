# Unraid + Immich Design

This app can become an Unraid Community App as a Docker container with a Web UI. The container should run all matching locally on the Unraid server and should not upload images, hashes, or metadata to an external service.

## Primary Goal

Match finished images from one Immich user account against RAW originals from another Immich user account, assign a confidence score, provide a dry-run review workflow, and then produce a safe action plan for matched RAW files.

## Compatibility Rules

Immich is the system of record for assets it manages. The app must not move, rename, or delete files inside Immich-managed `upload` or `library` folders directly through the filesystem. Immich stores asset paths and metadata in Postgres, and filesystem-only changes can leave Immich with missing or untracked assets.

Allowed:

- Read Immich data through the API.
- Read Immich Postgres using read-only queries for performance.
- Read mounted image files to compute local hashes and previews.
- Write the app's own cache, manifests, and dry-run reports under the appdata volume.
- Apply review tags through the Immich API when the user confirms the tag plan.

Forbidden:

- Updating Immich database rows directly.
- Moving files out of Immich `upload`, `library`, `thumbs`, `encoded-video`, or `profile` folders.
- Deleting Immich assets by filesystem path.
- Assuming a fixed database schema without checking the Immich version and adapter compatibility.

## Recommended Container Mounts

```text
/config              App config, cache, dry-run manifests
/immich-upload:ro    Immich upload/library root, read-only
```

The app should not require a writable media mount for the Immich workflow. Deletion should happen later in Immich after the user filters and reviews the `not used` tag.

## Immich Integration

Use an admin-level Immich API key for discovery where possible, then filter assets by owner:

- RAW account: source user for `.cr2`, `.cr3`, `.arw`, `.dng`, `.nef`, etc.
- Final account: source user for `.jpg`, `.jpeg`, `.png`.

If the API cannot efficiently return all required metadata, use Postgres as a read-only accelerator:

- Connect with a read-only database user.
- Detect Immich server/database version at startup.
- Use versioned query adapters.
- Fail closed to API-only mode if the schema is unknown.
- Never write to Postgres.

The matching model should keep Immich asset IDs alongside filesystem paths so review results remain stable even if a filename changes.

## Matching Signals

- Exact or normalized filename stem.
- Shared trailing image number.
- Capture timestamp.
- Camera make/model.
- Immich metadata such as original filename, original path, local date/time, checksum, and owner where available.
- Local perceptual hash from finished image and RAW embedded preview.
- Local thumbnails/previews already generated by Immich, only if the mounted paths and permissions make that safe and compatible.

## Workflow

1. Configure Immich URL and API key.
2. Select RAW Immich user.
3. Select FINAL Immich user.
4. Configure optional filesystem mounts.
5. Run scan.
6. Save dry-run manifest.
7. Review low-confidence matches.
8. Re-run dry-run report.
9. Apply `RAW Found` / `No RAW` tags to final-account assets and `Keeper` / `not used` tags to RAW-account assets through the Immich API.
10. Delete or archive `not used` RAWs in Immich after review.

## Preferred Immich Culling Workflow

For RAW files managed by Immich, prefer tagging over moving. This keeps deletion and asset tracking inside Immich.

Tags to apply to final-account assets:

- `RAW Found`: final image has a matched RAW.
- `No RAW`: final image does not have a matched RAW.

Tags to apply to RAW-account assets:

- `Keeper`: RAW file has a matched final image.
- `not used`: RAW file has no matched final image.

The app should apply these through the Immich API when supported. Direct database writes are forbidden. If API tag mutation is not available or not stable for the installed Immich version, the fallback is writing XMP sidecars and asking Immich to run its sidecar discover/sync metadata jobs.

Do not edit RAW EXIF metadata in-place. RAW formats are proprietary enough that in-place metadata writes are higher risk than sidecar files. XMP sidecars are the safer metadata path because Immich supports reading tags, rating, description, date/time, and location from sidecars.

Deletion should happen in Immich after review:

1. Run match.
2. Apply dry-run tags only.
3. Review low-confidence matches in this app.
4. Apply `RAW Found` / `No RAW` to finals and `Keeper` / `not used` to RAWs.
5. In Immich, filter/tag-view `not used` assets and delete from the RAW account.

## Unraid Community App Shape

- Docker image: `photo-culling-assistant`
- Web UI port: configurable, default `8356`
- Appdata path: `/mnt/user/appdata/photo-culling-assistant`
- Template variables:
  - Immich URL
  - Immich API key
  - RAW user selector or user ID
  - Final user selector or user ID
  - Immich media mount path
  - Dry-run default: enabled

## Safety Defaults

- Dry-run is enabled by default.
- Tag application is disabled until a dry-run manifest exists.
- File move and delete operations are not part of the Unraid workflow.
- Every tag application writes CSV and JSON manifests.
- The app should show the exact Immich asset IDs and paths before applying tags.
- Database integration is read-only and optional.
